# Transitions

A transition defines how the leaving and entering pages animate. It is an object that implements the `TransitionDef` interface — three functions that run at different points in the navigation lifecycle.

---

## The `TransitionDef` interface

```ts
interface TransitionDef {
  /**
   * Called synchronously in useLayoutEffect before the browser paints.
   * Use this to set the initial hidden state of the incoming page so
   * it doesn't flash visible before the entry animation fires.
   */
  setInitialState(content: HTMLElement): void

  /**
   * Called when the user clicks a link. Runs on the ghost clone of the
   * leaving page. Must return a gsap.core.Tween so the system can kill
   * it if an aborted transition needs to be cleaned up.
   */
  exit(ghost: HTMLElement): gsap.core.Tween

  /**
   * Adds the entry animation to the provided timeline. Called after
   * setInitialState, before the page's usePageEntry animations.
   */
  entry(tl: gsap.core.Timeline, content: HTMLElement): void
}
```

`content` in `setInitialState` and `entry` refers to the `.page-content` element of the **incoming** page. `ghost` in `exit` is a fixed-position clone of the **leaving** page.

## Built-in transitions

### `defaultTransition`

Included by `init` in `src/motion/transitions/default.ts`.

- **Exit:** the ghost scales down to 0.8, moves up 30% of the viewport, and fades out
- **Entry:** the incoming page clips in from the top (100svh → 0), revealing top-to-bottom

Both animations run with `expo.inOut` easing over 1.2 seconds and run in parallel.

```ts
import { defaultTransition } from './transitions/default'

defineMotionConfig({ default: defaultTransition })
```

### `fadeTransition`

Available via `view-motion add fade`. Adds `src/motion/transitions/fade.ts`.

- **Exit:** the ghost fades out
- **Entry:** the incoming page fades in

Simple and works well for most transitions.

```sh
view-motion add fade
```

```ts
import { fadeTransition } from './transitions/fade'

defineMotionConfig({ default: fadeTransition })
```

## How transitions are selected

When navigating, the system resolves the transition to use by:

1. Looking up the **namespace** for the current path and the target path via `motionConfig.namespaces`
2. Checking `motionConfig.transitions` for an exact match: `'fromNs -> toNs'`
3. Checking for a wildcard match: `'* -> toNs'`
4. Falling back to `motionConfig.default`

```ts
defineMotionConfig({
  default: defaultTransition,
  namespaces: {
    '/': 'home',
    '/about': 'about',
    '/work': 'work',
  },
  transitions: {
    // Navigating from home to about: fade
    'home -> about': fadeTransition,
    // Navigating to work from anywhere: default
    '* -> work': defaultTransition,
  },
})
```

If you don't configure namespaces, every route gets the namespace `"default"` and all transitions fall back to `motionConfig.default`.

## Timing

The exit and entry animations run in parallel:

```
t=0        Click. Ghost created. Exit tween starts. router.push() called.
           React starts rendering new page.
t=~0       New page commits. useLayoutEffect: setInitialState hides content.
           useEffect: entry tween starts.
t=0.4s     usePageEntry animations begin (default pageAnimDelay).
t=1.2s     Clip-in completes. Ghost exit finishes. Ghost removed.
```

The entry and exit both have a 1.2s duration in the default transition. Because they start at essentially the same time, they overlap completely.

---

## Using transitions per-pair

```ts
defineMotionConfig({
  default: defaultTransition,
  namespaces: {
    '/': 'home',
    '/about': 'about',
    '/contact': 'contact',
  },
  transitions: {
    // Only when going from home → about
    'home -> about': fadeTransition,

    // Wildcard: going TO contact from anywhere
    '* -> contact': fadeTransition,
  },
})
```

<Callout>
  Transition keys use namespace strings, not raw paths. Make sure the namespaces you reference in `transitions` are defined in the `namespaces` map.
</Callout>
